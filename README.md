# Vesta .NET — Serviço de Apoio à Decisão

**Global Solution FIAP 2026/1 — Advanced Business Development with .NET**

---

## O que é este serviço

A plataforma **Vesta** gerencia abrigos emergenciais em situações de desastre. O backend principal é em **Java/Spring** e cuida de todo o cadastro operacional. Este serviço **.NET é auxiliar**: ele lê os dados que o Java gravou e responde a pergunta central dos gestores:

> *"Quais abrigos precisam de atenção agora, e por quê?"*

O serviço calcula um **score de criticidade** (0–100) para cada abrigo, gera um **ranking de priorização**, produz **justificativas explicáveis** e **recomendações de ação**, e persiste o histórico de cada análise para rastreabilidade.

---

## Arquitetura

Clean Architecture com 4 projetos separados. As dependências sempre apontam para dentro (Domain no centro):

```
VestaNET/
├── src/
│   ├── VestaNET.Domain          ← regras de negócio puras, sem dependências externas
│   ├── VestaNET.Application     ← casos de uso, interfaces, DTOs
│   ├── VestaNET.Infrastructure  ← EF Core/Oracle, repositórios, JWT
│   └── VestaNET.Api             ← controllers, Swagger, Health Checks
└── tests/
    └── VestaNET.Tests           ← testes unitários xUnit
```

## Política de não-conflito com o Java

| Tabela | Dono | Este serviço |
|---|---|---|
| `TB_ABRIGO`, `TB_ESTOQUE_ABRIGO`, `TB_RECURSO`, `TB_OCORRENCIA`, `TB_SOLICITACAO_RECURSO` | Java / Disciplina de Banco | Somente lê |
| `TB_NET_ANALISE`, `TB_NET_RECOMENDACAO` | **Este serviço** | Cria e grava |

O prefixo `TB_NET_` garante zero colisão de nomes com o Java. As tabelas compartilhadas são mapeadas com `ExcludeFromMigrations()` — o EF as lê mas **nunca** cria, altera ou dropa.

## Como o score de criticidade é calculado

O motor `CalculadoraCriticidade` combina **4 fatores** lidos diretamente do banco Oracle:

### Fator 1 — Ocupação (peso 40%)

**Fonte:** `TB_ABRIGO.QT_OCUPACAO_ATUAL` e `TB_ABRIGO.QT_CAPACIDADE_MAXIMA`

| Taxa de ocupação | Score bruto |
|---|---|
| ≥ 100% | 100 |
| ≥ 90% | 85 |
| ≥ 75% | 60 |
| ≥ 50% | 30 |
| < 50% | 0 |

### Fator 2 — Recursos abaixo do mínimo (peso 25%)

**Fonte:** `TB_ESTOQUE_ABRIGO` — conta quantos registros têm `QT_ATUAL < QT_MINIMA`

Score bruto = (recursos críticos / total de recursos) × 100

### Fator 3 — Ocorrências abertas por severidade (peso 25%)

**Fonte:** `TB_OCORRENCIA` — registros com `ST_STATUS ≠ 'RESOLVIDA'`

| Severidade | Pontos |
|---|---|
| CRÍTICA | 40 pts |
| ALTA | 25 pts |
| MÉDIA | 12 pts |
| BAIXA | 5 pts |

Score bruto = soma dos pontos (máximo 100)

### Fator 4 — Solicitações pendentes (peso 10%)

**Fonte:** `TB_SOLICITACAO_RECURSO` — registros com `ST_STATUS IN ('ABERTA', 'EM_ANALISE', 'EM_ATENDIMENTO')`

Score bruto = quantidade de pendentes × 25 (máximo 100)

### Fórmula final

```
Score = (F1 × 0,40) + (F2 × 0,25) + (F3 × 0,25) + (F4 × 0,10)
```

### Faixas de nível

| Score | Nível |
|---|---|
| ≥ 75 | **CRÍTICA** — atenção imediata, prioridade máxima |
| ≥ 50 | **ALTA** — intervenção necessária em breve |
| ≥ 25 | **MÉDIA** — monitoramento ativo recomendado |
| < 25 | **BAIXA** — dentro dos parâmetros normais |

```sql
    CONSTRAINT FK_NET_REC_ANALISE FOREIGN KEY (ID_ANALISE)
        REFERENCES TB_NET_ANALISE(ID_ANALISE) ON DELETE CASCADE
);

COMMIT;
```

## Como testar — passo a passo

### Via Dashboard (`http://localhost:5050`)

1. Clique em **Conectar** — carrega os dados do Oracle automaticamente
2. Aba **Abrigos** — lista todos os abrigos com indicadores
3. Aba **Ranking** — abrigos ordenados por criticidade
4. Clique em qualquer abrigo → vai para **Análise**
5. Clique em **Analisar** — executa e exibe o score completo com detalhes do cálculo
6. Clique em **Histórico** — vê análises anteriores do mesmo abrigo

### Via Swagger (`http://localhost:5050/swagger`)

**1. Obter token:**
```

```
Copie o `token` → clique **Authorize 🔓** → cole → **Authorize**

**2. Ver abrigos:**
```

```

**3. Analisar criticidade:**
```
POST /api/analises/abrigos/{id}
```
Retorna score, nível, explicação, parâmetros lidos do banco, detalhes de cada fator e recomendações.

**4. Ranking:**
```
GET /api/analises/ranking

```

**5. Histórico:**
```
GET /api/analises/abrigos/{id}/historico
```

**6. Health Checks:**
```
GET /health     → liveness da aplicação

```

## Consultas SQL para verificar os dados do cálculo

### Ocupação de todos os abrigos
```sql
SELECT
    id_abrigo,
    nm_abrigo,
    qt_ocupacao_atual,
    qt_capacidade_maxima,
    ROUND(qt_ocupacao_atual / qt_capacidade_maxima * 100, 1) AS taxa_pct,
    st_status
FROM TB_ABRIGO
ORDER BY taxa_pct DESC;
```

### Estoque por abrigo (com status crítico)
```sql
SELECT
    a.nm_abrigo,
    r.nm_recurso,
    r.tp_recurso,
    e.qt_atual,
    e.qt_minima,
    CASE WHEN e.qt_atual < e.qt_minima THEN 'CRÍTICO ⚠' ELSE 'OK' END AS situacao,
    e.dt_atualizacao
FROM TB_ESTOQUE_ABRIGO e
JOIN TB_ABRIGO  a ON a.id_abrigo  = e.id_abrigo
JOIN TB_RECURSO r ON r.id_recurso = e.id_recurso
ORDER BY a.nm_abrigo, situacao DESC;
```

### Apenas recursos críticos (abaixo do mínimo)
```sql
SELECT
    a.nm_abrigo,
    r.nm_recurso,
    e.qt_atual,
    e.qt_minima,
    e.qt_minima - e.qt_atual AS faltando
FROM TB_ESTOQUE_ABRIGO e
JOIN TB_ABRIGO  a ON a.id_abrigo  = e.id_abrigo
JOIN TB_RECURSO r ON r.id_recurso = e.id_recurso
WHERE e.qt_atual < e.qt_minima
ORDER BY faltando DESC;
```

### Ocorrências abertas por abrigo
```sql
SELECT
    a.nm_abrigo,
    o.nm_titulo,
    o.tp_severidade,
    o.st_status,
    o.dt_ocorrencia
FROM TB_OCORRENCIA o
JOIN TB_ABRIGO a ON a.id_abrigo = o.id_abrigo
WHERE o.st_status != 'RESOLVIDA'
ORDER BY a.nm_abrigo,
    CASE o.tp_severidade
        WHEN 'CRITICA' THEN 1
        WHEN 'ALTA'    THEN 2
        WHEN 'MEDIA'   THEN 3
        ELSE 4
    END;
```

### Solicitações pendentes por abrigo
```sql
SELECT
    a.nm_abrigo,
    r.nm_recurso,
    s.qt_solicitada,
    s.st_status,
    s.ds_justificativa,
    s.dt_solicitacao
FROM TB_SOLICITACAO_RECURSO s
JOIN TB_ABRIGO  a ON a.id_abrigo  = s.id_abrigo
JOIN TB_RECURSO r ON r.id_recurso = s.id_recurso
WHERE s.st_status IN ('ABERTA', 'EM_ANALISE', 'EM_ATENDIMENTO')
ORDER BY a.nm_abrigo, s.dt_solicitacao;
```

### Visão completa — todos os fatores de um abrigo de uma vez
```sql
SELECT
    a.nm_abrigo,
    ROUND(a.qt_ocupacao_atual / a.qt_capacidade_maxima * 100, 1) AS ocupacao_pct,
    (SELECT COUNT(*) FROM TB_ESTOQUE_ABRIGO e
     WHERE e.id_abrigo = a.id_abrigo AND e.qt_atual < e.qt_minima)  AS recursos_criticos,
    (SELECT COUNT(*) FROM TB_OCORRENCIA o
     WHERE o.id_abrigo = a.id_abrigo AND o.st_status != 'RESOLVIDA') AS ocorrencias_abertas,
    (SELECT COUNT(*) FROM TB_SOLICITACAO_RECURSO s
     WHERE s.id_abrigo = a.id_abrigo
       AND s.st_status IN ('ABERTA','EM_ANALISE','EM_ATENDIMENTO'))  AS solicitacoes_pendentes
FROM TB_ABRIGO a
ORDER BY ocupacao_pct DESC;
```

### Histórico de análises geradas pelo serviço .NET
```sql
SELECT
    n.id_analise,
    a.nm_abrigo,
    n.vl_score,
    n.tp_nivel,
    n.ds_justificativa,
    n.dt_analise
FROM TB_NET_ANALISE n
JOIN TB_ABRIGO a ON a.id_abrigo = n.id_abrigo
ORDER BY n.dt_analise DESC;
```
